## 치명적 버그 (계산 결과가 틀림)

### BUG-01. 매수 페이지 정책상품 대출한도 항상 0원 표시

- **파일**: `app/buy/page.tsx:120-136`
- **현상**: 정책대출상품(디딤돌, 신생아특례) 자격 충족 시 "최대 0원"으로 표시
- **원인**: `evaluateRentProduct()`에 `deposit: 0`을 전달함
  ```ts
  // buy/page.tsx:126
  deposit: 0, // ← 전세보증금이 아니라 매매용인데 0을 넣음
  ```
  - `calc-rent.ts:65`: `byRatio = floor(0 × ratioMax) = 0`
  - `maxLoanAmount = min(0, maxAmount) = 0`
- **재현**: 매수 페이지 → 무주택 + 연소득 5000만원 → 디딤돌대출 "자격 충족" → "최대 0원"
- **영향**: 모든 매수 정책상품의 대출한도가 0원으로 표시됨. 사용자에게 잘못된 정보 제공

### BUG-02. 중개수수료 데이터 오류 (6억 초과 구간)

- **파일**: `public/data/tax_rules.json:61-67`
- **현상**: 6억 초과 매물의 중개수수료가 실제 법정요율과 다르게 계산됨
- **현재 데이터**:
  ```
  2억~6억: 0.4% ← 정확
  6억~9억: 0.5% ← 오류 (실제: 0.4%)
  9억 초과: 0.9% ← 오류 (실제: 9~12억 0.5%, 12~15억 0.6%, 15억+ 0.7%)
  ```
- **실제 법정요율 (2021.10 개편)**:
  ```
  5천만 미만: 0.6% (한도 25만)
  5천만~2억: 0.5% (한도 80만)
  2억~9억:   0.4% (한도 없음)   ← 6억~9억도 0.4%여야 함
  9억~12억:  0.5%
  12억~15억: 0.6%
  15억 이상: 0.7%
  ```
- **재현**: 양도세 페이지 → 양도가 7억 입력 → 중개수수료 350만원(0.5%) 표시 → 실제는 280만원(0.4%)
- **참고**: `app/guide/page.tsx:249-264`의 가이드 테이블은 올바른 6단계 요율을 표시하고 있어서, 가이드와 실제 계산이 불일치

## 주요 버그 (기능 누락 / 데이터 미사용)

### BUG-03. 전세 페이지: 나이, 혼인 기간 입력이 계산에 반영 안 됨

- **파일**: `app/rent/page.tsx:27-28, 123-141` / `lib/calc-rent.ts`
- **현상**: 나이, 혼인 기간 폼 필드가 존재하지만 `evaluateRentProduct()`에 전달되지 않음
- **영향**:
  - **나이**: 청년전용 버팀목(만 19~34세)에 40세를 입력해도 자격 충족으로 나옴
  - **혼인 기간**: 신혼부부 전용(혼인 7년 이내) 상품에 15년을 입력해도 자격 충족으로 나옴
  - `loan_products.json`의 `ageRange: {min: 19, max: 34}`도 로직에서 미참조
- **재현**: 전세 페이지 → 나이 40세 + 가구특성 "청년" → "청년전용 버팀목전세자금" 자격 충족

### BUG-04. 매수 페이지: Select 컴포넌트 도입으로 인한 UI 불일치

- **파일**: `app/buy/page.tsx:13, 221-234` / `components/ui/select.tsx`
- **현상**: 매수 페이지의 "가구 특성"만 Radix Select(팝업 드롭다운), 전세 페이지는 NativeSelect(기본 브라우저 select)
- **마이그레이션 계획과 불일치**: 계획에서 "shadcn Select 안 씀 — Radix Select는 register() 불가"로 결정했으나 buy에만 도입됨
- **동작은 하지만** (`setValue` 사용으로 우회), 두 페이지 간 같은 필드의 UI가 다름

## 엣지 케이스

### EDGE-01. 입력값 전체 삭제 시 NaN 전파

- **재현**: 아무 숫자 입력 필드 → 전체 삭제 (빈 칸)
- **동작**: `valueAsNumber: true` → NaN 반환 → `Number(NaN) || 0` → 0으로 처리
- **결과**: 정상 작동 (0으로 fallback)

### EDGE-02. 음수 입력

- **재현**: "보유 현금" -5000 입력
- **동작**: 별도 min 제약 없음 → `futureAsset`이 음수 가능 → 매수가능가 음수
- **결과**: 음수 금액이 ResultCard에 그대로 표시됨 (formatKRW는 음수 처리 가능하지만 `-5,000만 원` 같은 형태로 나옴)
- **심각도**: 낮음 (사용자가 음수를 입력할 이유가 없지만, 방어 로직 없음)

### EDGE-03. 양도차익이 0 이하일 때 실수령액 표시

- **재현**: 양도세 페이지 → 취득가 50000, 양도가 30000
- **동작**: gain = -2억 → taxableGain=0, totalTax=0, 중개수수료만 차감
- **결과**: 실수령액 배너와 "양도차익이 없어요!" 카드가 동시에 표시됨 — 정상

### EDGE-04. 양도세 페이지 — 필요경비가 양도차익보다 클 때

- **재현**: 취득가 30000, 양도가 35000, 경비 10000
- **동작**: gain = 5억 - 3억 - 1억 = -5000만 → gain ≤ 0 → 세금 0
- **결과**: 정상 ("양도차익이 없어요!")

### EDGE-05. 조정대상지역 + 2주택 + 생애최초

- **재현**: 매수 페이지 → 조정대상지역 + 2주택+ + 생애최초 ON
- **동작**: `calcLtv`에서 `housingCount: min(2,2)=2`로 검색 → `isFirstTime=true`인 규칙이 없음 → fallback으로 `isFirstTime=false + housingCount=2` → LTV 0%
- **결과**: 정상 (2주택 이상은 생애최초 혜택 없음)

## 코드 품질 이슈 (기능에는 영향 없음)

### CODE-01. `targetYears` Nullish 연산자 불일치

- **파일**: `app/buy/page.tsx:112`
- **코드**: `targetYears: Number(v.targetYears) ?? 0`
- **문제**: 다른 필드는 모두 `|| 0` 패턴인데 이것만 `?? 0`
- `??`는 NaN을 잡지 못함 (NaN은 null/undefined가 아님)
- 라디오 버튼이라 NaN이 불가능해서 실제 영향은 없지만 패턴 불일치

### CODE-02. 전세 페이지 useMemo 과도한 의존성

- **파일**: `app/rent/page.tsx:84`
- **코드**: `[v, products]` — `v`는 전체 폼 상태 객체
- `age`, `marriageYears` 변경 시에도 불필요하게 재계산됨
- 성능 이슈는 미미하지만, 의존성을 명시적으로 나열하는 것이 좋음

### CODE-03. 매수 페이지에서 `evaluateRentProduct` 함수명 혼동

- **파일**: `app/buy/page.tsx:124`
- 전세용 함수(`evaluateRentProduct`)를 매수 정책상품 평가에 재사용
- 기능상 동작하나 함수명이 의도와 맞지 않음 (→ `evaluateProduct`로 리네임하는 게 명확)
